Real-Time Shipment Tracker

PART 1 — Use Case & Architecture ~8 minutes

Opening (30 sec)

"Hello everyone. My name is David Fellner and my project is a Real-Time Shipment Tracker — a fully serverless AWS pipeline that ingests live GPS events from delivery vehicles, processes them in real time, and displays them on a live map dashboard."

Problem & Use Case (1.5 min)

"Imagine you run a logistics company with 50 trucks driving across Europe. Your customers want to know: where is my shipment right now, and will it arrive on time?

The challenge: GPS devices send position updates every few seconds — thousands of events per minute. You need infrastructure that:
  • scales as your fleet grows,
  • loses no data, even if a device briefly disconnects,
  • is cheap — you don't want to run servers 24/7 for a fluctuating workload,
  • and reacts in real time, not with a 10-minute delay.
That's exactly what this system solves."

Architecture — Service by Service (5 min)

"Let me walk through the architecture. Every service choice has a concrete reason."
Python Simulator → AWS IoT Core → Amazon Kinesis Data Streams ↓ AWS Lambda (process_event) → Amazon DynamoDB ↓ API Gateway → AWS Lambda (get_shipments) → React Dashboard (Amplify)

AWS IoT Core

"GPS devices connect via MQTT — the standard IoT protocol, ultra-lightweight and resilient on poor connections. I chose IoT Core over sending directly to Kinesis because IoT Core provides certificate-based device authentication — each device has its own TLS certificate, no passwords or API keys that could be stolen. The built-in Rules Engine forwards MQTT messages to Kinesis with zero custom code."

Amazon Kinesis Data Streams

"Events land in Kinesis — an ordered, replayable data stream. Why not SQS? SQS is a queue for individual tasks. Kinesis is better for continuous high-volume data streams like GPS updates. Key advantage: if Lambda fails, Kinesis replays the events automatically — no data loss. I use On-Demand mode, so it scales automatically with traffic."

AWS Lambda — process_event

"Kinesis triggers a Lambda function that processes each GPS event: parses coordinates, calculates status — on time, delayed, arriving soon — and writes the result to DynamoDB. Why Lambda over EC2? No persistent compute needed. Lambda runs only when events arrive and I pay per invocation only. Perfect for spiky, event-driven workloads."

Amazon DynamoDB

"Processed shipment data is stored in DynamoDB, keyed by shipmentId. Each GPS update overwrites the current position. Why not RDS? My access pattern is simple — fetch by ID or fetch all — no joins, no SQL needed. DynamoDB gives me single-digit millisecond latency and is also serverless."

API Gateway + React Dashboard

"The frontend polls a REST endpoint via API Gateway and a second Lambda every few seconds. The React dashboard with Leaflet maps renders each shipment as a colored pulsing marker — cyan for in-transit, green for on-time, red for delayed. Hosted on AWS Amplify for automatic HTTPS and CDN delivery."

Well-Architected Pillars (1.5 min)

Cost Optimization

"The entire stack is serverless — I pay exactly proportional to usage. No idle servers. For this PoC the total cost is under $5, even with hours of testing. Kinesis On-Demand, Lambda per-invocation pricing, DynamoDB On-Demand — all elastic."

Security

"Three layers: First, IoT Core uses X.509 certificate authentication — no device connects without a valid certificate. Second, IAM Least Privilege — the processing Lambda can only write to DynamoDB, nothing else. Third, no hardcoded credentials — all endpoints and keys come from CDK outputs and environment variables, never in source code."

PART 3 — Q&A Preparation ~5 minutes
Why Kinesis and not SQS?
SQS delivers each message once and deletes it. Kinesis is a stream — ordered, replayable, supports multiple consumers reading independently. For continuous GPS data at scale, Kinesis is the right tool.
Why not WebSockets instead of polling?
WebSockets would be the production upgrade — AWS API Gateway supports WebSocket APIs natively. For this PoC, polling every 5 seconds is sufficient for GPS update frequency and much simpler to implement. I'd switch for production.
What happens if Lambda crashes?
Kinesis retains events for up to 7 days and retries failed Lambda batches automatically. No data is lost — it's built into the Kinesis-Lambda integration.
Could this scale to 5,000 trucks?
Yes, without architecture changes. Kinesis On-Demand handles ingestion scaling, Lambda scales per shard, DynamoDB On-Demand scales automatically. The architecture is designed for this.
Why DynamoDB over Timestream?
Timestream would be better for querying historical routes. My requirement is only the current state of each shipment — no history queries needed. DynamoDB is cheaper and simpler for this access pattern.
